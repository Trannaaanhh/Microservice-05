1. Problem Overview

The original bookstore microservice platform had basic service separation but lacked a strong control layer for identity management, role authorization, distributed failure handling, and measurable operational evidence.

This upgrade addresses those gaps through five major directions.
First, token based identity is standardized through the auth service.
Second, role based access control is enforced at the gateway level.
Third, order processing uses a saga orchestration model with compensation logic.
Fourth, service to service reservation flow uses RabbitMQ request reply messaging.
Fifth, health and metrics endpoints plus repeatable scripts provide validation evidence.

2. Scope and Technical Objectives

2.1 Implementation Scope

The implementation scope includes the following concrete areas.
Auth service token issue and token verification with embedded role claim.
Gateway middleware pipeline for authentication, request rate limiting, and request logging.
Order service orchestration flow for payment reservation and shipping reservation with compensation.
RabbitMQ consumers inside payment service and shipping service.
Metrics endpoints in key services for runtime snapshots.
Validation scripts for fault simulation, load testing, and deterministic data seeding.

2.2 Evaluation Objectives

The technical evaluation objective is to verify the system in a measurable way.
Token issuance and verification must work correctly.
Role based path protection must behave correctly for customer and staff.
Saga state transition must match expected branch behavior in failure and success scenarios.
Compensation logic must run when shipping reservation fails after payment reservation success.
Gateway health endpoint must remain stable under repeated requests.
All validation must be reproducible with provided scripts.

3. System Architecture After Upgrade

3.1 Core Components

The upgraded system is deployed with Docker Compose.
Shared infrastructure includes PostgreSQL and RabbitMQ.
API Gateway runs on port 8000 as the external entry point.
Auth Service runs on port 8010 and provides token operations.
Domain services run on ports 8001 to 8009 and provide business capabilities.
RabbitMQ supports request reply messaging for reservation workflow.
PostgreSQL uses per service databases for data isolation.

3.2 High Level Control Flow

A typical checkout flow is executed as follows.
Step 1 User logs in through API Gateway.
Step 2 Gateway requests token from auth service and stores access token in session.
Step 3 Gateway validates token in middleware for protected routes.
Step 4 Checkout action triggers order creation in order service.
Step 5 Order service requests payment reservation through RabbitMQ.
Step 6 Order service requests shipping reservation through RabbitMQ.
Step 7 If shipping reservation fails then order service requests payment release as compensation.
Step 8 Final order state is persisted and emitted as domain event.

4. Detailed Technical Implementation

4.1 Auth Service JWT and RBAC Internals

Auth service issues access token using HS256 signing.
Token fields include subject role issue time and expiration time.
Token duration is configured by environment variable for access token seconds.
Token verification endpoint validates signature and expiration.
Service also exposes health and metrics endpoints.

RBAC logic is enforced at gateway side through Django user groups.
Supported roles are customer and staff.
User is created automatically in demo flow during first login.
Gateway checks role before allowing sensitive operations.
Staff only paths are blocked when the user role is not staff.

4.2 Gateway Middleware Pipeline

Gateway middleware chain includes three custom stages.
JWT authentication middleware protects non exempt paths.
Rate limit middleware enforces request quota per client address per minute.
Request logging middleware records method path status duration and source address.

JWT middleware behavior summary.
It accepts bearer token from authorization header or session token from login flow.
It decodes claims using shared secret configuration.
When token is missing invalid or expired then access is rejected and stale session token is removed.

Rate limit middleware behavior summary.
It uses a sliding sixty second window keyed by remote address.
The default limit is one hundred twenty requests per minute.
Overflow returns status code four hundred twenty nine.

Request logging middleware behavior summary.
It measures response time for each request.
It emits structured log payload and updates service counters.

4.3 Saga Orchestration Design in Order Service

Order service implements saga orchestration in order creation flow.
Order state model has business status and saga state plus failure reason.

Business status values are pending confirmed and failed.
Saga state values are order pending payment reserve failed shipping reserve failed compensated and confirmed.

Execution sequence.
Step 1 Create order with pending status and order pending saga state.
Step 2 Call payment reserve over RabbitMQ request reply.
Step 3 If payment reserve fails then mark order failed and set payment reserve failed state.
Step 4 If payment succeeds then call shipping reserve.
Step 5 If shipping reserve fails then call payment release as compensation and mark shipping reserve failed compensated state.
Step 6 If both reservations succeed then set order confirmed state.

Each branch returns final order state for simple gateway rendering and auditability.

4.4 RabbitMQ Integration and Consumer Model

Order service provides two messaging utilities.
One utility performs synchronous request reply call with correlation id and callback queue.
Another utility publishes event envelope to fanout exchange for domain events.

Payment and shipping services run consumer loops in background threads.
Payment service listens to payment reserve and payment release queues.
Shipping service listens to shipping reserve and shipping release queues.

Each consumer does the following sequence.
Read message and parse payload.
Apply simulated failure condition when configured.
Update domain record in database.
Send reply with success or error payload.
Acknowledge original message.

Both consumers use prefetch one to process one message at a time.
Both consumers retry connection in a loop when broker errors occur.

4.5 Observability and Metrics Model

Observability model is lightweight and service local.
Gateway tracks total requests and response count by status code.
Auth service tracks total requests and total issued tokens.
Order service tracks total orders confirmed orders and failed orders.

All key services provide health endpoint and metrics endpoint.
This makes smoke checks and script based evidence collection straightforward.

6. Validation Results and Evidence

6.1 Acceptance Summary Clean Build

Current evidence indicates successful acceptance for target scope.
Payment failure scenario ends with failed status and payment reserve failed saga state.
Shipping failure scenario ends with failed status and shipping reserve failed compensated saga state.
Success scenario ends with confirmed status and confirmed saga state.
Gateway health load runs complete without failed requests in recorded artifacts.

6.2 Quantitative Evidence Snapshot

From project artifacts two representative snapshots are available.
Snapshot one from fault and load testing report shows five hundred requests with zero failures and stable latency percentiles.
Snapshot two from load result artifact shows fifty requests with zero failures and average latency around fifty three point six milliseconds.

These outcomes support baseline stability under sequential synthetic load.

Step 5 Execute load test with default request count.
Command powershell -ExecutionPolicy Bypass -File .\scripts\load-test.ps1

Step 6 Execute load test with custom request count.
Command powershell -ExecutionPolicy Bypass -File .\scripts\load-test.ps1 -Requests 500

7.3 Fault Simulation Method

Target endpoint is http colon slash slash localhost colon 8004 slash orders slash
Payload field simulate failure step controls scenario.
Value payment triggers payment reservation failure branch.
Value shipping triggers shipping reservation failure branch and payment compensation.
No field triggers normal success branch.

Expected output is validated by order status and saga state values.
Compensation branch should show payment release result.

8. Academic Assessment and Discussion

8.1 Pass Fail Assessment

The upgraded system satisfies core academic objectives.
Identity and role control are implemented and testable.
Distributed failure handling with compensation is implemented and testable.
Validation evidence is script based and reproducible.

8.2 Architectural Contributions

The upgrade contributes a clear separation of concerns.
Gateway handles access control and frontend integration.
Domain services preserve ownership of business state.
Order service coordinates distributed steps without direct database coupling.
RabbitMQ enables loose coupling while preserving deterministic flow.

8.3 Limitations and Non Blocking Debt

Current limitations remain and are non blocking for this release.
System does not yet include centralized distributed tracing.
Metrics are not yet exported to a dedicated monitoring stack.
Current load script is sequential and does not represent concurrent stress.
Consumer startup still follows application server startup model.

9. Final Conclusion

This project upgrade provides a coherent and defensible microservice architecture for academic evaluation.
It combines token based identity, role based access control, saga orchestration with compensation, RabbitMQ integration, and reproducible validation evidence.
The system is now stronger in reliability, governance, and reportability compared with the original baseline.
